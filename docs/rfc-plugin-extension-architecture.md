# RFC：插件扩展架构

- 状态：提议中
- 最后更新：2026-07-10
- 目标版本：Plugin V1
- 决策负责人：LLM Space 维护者

## 问题陈述

LLM Space 已经支持多类外部能力：内置工具、MCP 工具、全局 Skills、跟踪源以及 Eve 项目调试。目前每种能力都通过一套定制组合进入产品，包括核心数据类型、Bun 进程管理器、渲染器客户端、RPC 请求、提示词变量分支和设置存储。

这使得接入每一种新的智能体生态都变得昂贵且有风险。Eve 清楚地说明了这个问题：为了支持一种本地项目格式，我们引入了 Eve 专用的 Tool 类型、Thread 上下文、RPC 系列、渲染器分发分支、Skill 查找路径和启动种子器。如果对另一个框架重复这种设计，核心契约会持续膨胀，并让彼此无关的集成相互产生依赖。

从插件开发者的角度看，LLM Space 没有用于贡献源导入器、作用域化工具或作用域化 Skills 的稳定 API。开发者必须修改应用本身，并理解桌面端的内部边界。从用户的角度看，各种集成无法以一致的方式启用、诊断或理解。

因此，LLM Space 需要一种插件架构，让受信任的扩展能够贡献调试能力，同时不在渲染器中运行扩展代码，也不必为每种集成新增一个应用专用分支。

## 解决方案

LLM Space 将引入一个受信任的 Bun 进程插件宿主和稳定的公开插件 API。插件通过静态清单描述自身，并通过明确的扩展点注册运行时处理器。宿主负责发现、校验、激活、生命周期管理、诊断、存储、贡献路由以及面向渲染器的 RPC 契约。

Plugin V1 将支持捆绑插件和明确受信任的本地开发插件，但不会声称能够沙箱化插件代码。插件运行时代码只会在桌面端 Bun 进程中执行；渲染器只会接收由宿主拥有的描述符、状态视图、诊断信息和标准化结果。

Eve 将成为第一个捆绑插件。`@llm-space/plugin-eve` 内部的 Eve 源码模块负责理解 Eve 项目，插件入口通过通用插件 API 注册 Eve 项目导入器、工具提供者、Skill 提供者和开发种子器。Eve 不再作为独立 workspace 包存在。

V1 的产品结果是：Eve 项目仍然可以被导入、检查和手动调用，但桌面端和核心路径将使用通用插件契约，而不是 Eve 专用路由。

## 用户故事

1. 作为智能体开发者，我希望 LLM Space 能够以插件形式加载对新智能体生态的支持，从而无需等待定制版应用发布就能调试该生态。
2. 作为 Eve 开发者，我希望迁移后 Eve 项目支持的行为与现在完全一致，从而采用插件不会中断我的本地调试工作流。
3. 作为 Eve 开发者，我希望项目的指令、工具和 Skills 继续限定在该项目范围内，从而全局 LLM Space 能力不会改变我正在调试的环境。
4. 作为 Eve 开发者，我希望能够手动调用导入的 Eve 工具，从而在继续模型运行前验证其参数和结果。
5. 作为 Eve 开发者，我希望新的 Eve 导入只生成通用插件 Thread，从而验证路径不会继续携带 Eve 专用持久化模型。
6. 作为 Eve 开发者，我希望现有的 Eve 开发环境变量继续生效，从而本地启动自动化不会在迁移期间中断。
7. 作为插件开发者，我希望有一个包含插件契约的稳定包，从而无需导入桌面端私有实现模块。
8. 作为插件开发者，我希望有一个包含稳定身份和兼容性元数据的静态清单，从而 LLM Space 能在执行插件前检查它。
9. 作为插件开发者，我希望注册源导入器，从而本地项目可以成为原生 LLM Space Thread。
10. 作为插件开发者，我希望注册工具提供者，从而导入的工具可以向模型暴露描述和 JSON Schema 参数。
11. 作为插件开发者，我希望通过宿主接收明确的手动工具调用，从而运行时可以在正确的项目上下文中执行它们。
12. 作为插件开发者，我希望注册 Skill 提供者，从而我的生态可以暴露自己的作用域化 Skills。
13. 作为插件开发者，我希望插件拥有的 Thread 上下文以 JSON 持久化，从而应用重启后调用仍能正确解析。
14. 作为插件开发者，我希望插件上下文数据带有版本，从而插件演进时可以迁移已持久化的状态。
15. 作为插件开发者，我希望拥有私有插件存储，从而运行时设置和缓存不会与其他插件冲突。
16. 作为插件开发者，我希望拥有结构化诊断，从而无需解析日志就能理解格式错误的项目和激活失败。
17. 作为插件开发者，我希望重复的贡献标识符被确定性地拒绝，从而路由不会依赖加载顺序。
18. 作为插件开发者，我希望长时间运行的操作能获得 AbortSignal，从而在支持的情况下用户取消可以传递到我的插件。
19. 作为插件开发者，我希望支持本地开发重载，从而无需重新安装桌面应用就能迭代。
20. 作为插件开发者，我希望有引擎兼容性检查，从而不兼容的 API 版本会在激活前失败。
21. 作为用户，我希望看到哪些插件是已捆绑、本地加载、活动、已禁用或失败的，从而了解当前有哪些能力可用。
22. 作为用户，我希望插件失败能够隔离，从而一个损坏的插件不会阻止 LLM Space 启动。
23. 作为用户，我希望缺少插件的 Thread 仍可检查，从而能够恢复或移除不可用的工具，而不会丢失提示词和消息数据。
24. 作为用户，我希望不可用的插件工具显示清晰的原因，从而不会把运行时缺失误认为模型失败。
25. 作为用户，我希望插件诊断信息不包含密钥和原始凭据，从而故障排查信息可以安全地查看和分享。
26. 作为用户，我希望 V1 中插件工具仍需手动执行，从而安装受信任的集成不会悄悄扩大自动执行范围。
27. 作为用户，我希望插件代码不进入渲染器，从而扩展无法直接操纵应用 UI 状态或 DOM 内容。
28. 作为维护者，我希望只有一套通用插件 RPC 系列，从而新增插件不会用生态专用请求扩大桌面端 RPC 契约。
29. 作为维护者，我希望只有一种通用的插件支持 Tool 类型，从而核心 Tool 联合类型不会随着每种集成增长。
30. 作为维护者，我希望只有一种通用的 Thread 插件上下文模型，从而生态来源和运行时作用域遵循一致的持久化契约。
31. 作为维护者，我希望源导入、Skill 查找和工具执行通过明确的注册表路由，从而所有权边界保持可测试。
32. 作为维护者，我希望插件清单可以在不激活插件的情况下校验，从而能够安全地发现打包和兼容性问题。
33. 作为维护者，我希望捆绑插件以静态方式声明，从而 Electrobun 打包能确定性地包含每个生产插件。
34. 作为维护者，我希望 Eve 专用持久化类型和路由被直接移除，从而新架构不会携带未要求的数据兼容层。
35. 作为维护者，我希望 Eve 项目解析与插件适配都归 `plugin-eve` 所有，同时保持内部模块边界，从而 fixture 和测试仍可直接验证 Eve 行为，而无需维护一个不会对外提供的独立包。
36. 作为维护者，我希望插件 API 变更遵循语义化版本，从而捆绑插件和本地插件拥有明确的兼容性契约。
37. 作为维护者，我希望插件尽可能延迟激活，从而未使用的集成不会增加启动成本或副作用。
38. 作为维护者，我希望启动种子器作为明确的贡献存在，从而由环境驱动的导入仍可被发现和诊断。
39. 作为维护者，我希望插件上下文迁移失败时不会修改原始 Thread 数据，从而用户文件仍可恢复。
40. 作为维护者，我希望 MCP 和跟踪集成在 Eve 迁移期间保持不变，从而 Plugin V1 保持为一个连贯的能力集合。

## 实现决策

### 产品指标

- 北极星指标是**首次获得插件支持的调试源所需时间**。
- 基线是当前的 Eve 集成。它能够工作，但需要 Eve 专用的核心类型、Thread 上下文、RPC、渲染器分发、提示词变量行为和启动代码。
- 当捆绑的 Eve 插件能够导入 Eve 项目、暴露项目作用域的 Tools 和 Skills，并通过通用插件路由完成一次明确的手动工具调用时，Plugin V1 即达成目标。
- 防护条件包括：全局 Skills 绝不泄漏到 Eve 环境；插件工具不会由模型循环自动执行；插件代码绝不在渲染器中运行；外部源项目不会被修改；诊断信息不会暴露密钥。

### 包和所有权边界

- 新增的 `@llm-space/plugin-api` 包将是编写插件所需的唯一公开依赖。
- 插件 API 包将包含可序列化的契约类型、清单 Schema、扩展点契约、宿主服务接口、结果和诊断类型，以及 `definePlugin` 等辅助函数。
- 插件 API 包不会导入桌面端代码，也不会暴露 Bun 进程实现对象。
- 桌面端 Bun 进程将拥有一个 `PluginManager`。它负责发现和校验清单、激活插件模块、维护贡献注册表、路由调用、暴露状态并隔离故障。
- 渲染器将通过类型化 RPC 使用由宿主拥有的插件客户端。插件不能定义任意 RPC 方法。
- `@llm-space/plugin-eve` 是 Eve 集成的唯一 workspace 包。其私有 `src/eve/` 模块检测项目、导入指令和 Skills、描述工具并执行 Eve 调用，包根入口负责插件 API 适配。
- Eve 领域实现不会作为独立包导出；插件适配层通过相对导入复用内部模块，不重复解析或执行逻辑。

### 支持的插件来源

- 生产捆绑包将以静态方式声明捆绑插件。必须采用静态声明，才能确保已打包的 Electrobun 构建以确定性方式包含清单和运行时入口。
- 可以从 `LLM_SPACE_PLUGIN_PATHS` 发现受信任的本地开发插件。多个路径使用平台的路径分隔符。
- 本地开发路径是一项明确的开发者功能。系统不会通过扫描任意用户目录来静默启用它们。
- 插件的启用状态和最近一次已知诊断将持久化到插件设置中。捆绑插件可以被禁用，但在 V1 中不能卸载。
- 远程安装、包下载和市场发现不在 V1 范围内。

### 静态清单契约

- 每个插件都必须提供一个 `llm-space.plugin.json` 清单。
- 在导入运行时模块前，必须先校验清单。
- 清单将包含稳定的插件 ID、显示名称、版本、运行时入口、LLM Space 引擎范围、API 版本、来源分类、声明的能力和贡献元数据。
- 插件 ID 使用稳定的点分标识符。`llm-space.eve` 保留给捆绑的 Eve 插件。
- 在同一个插件和扩展点类型内，贡献 ID 必须唯一。
- 清单中的贡献列表是描述性信息。运行时激活时必须准确注册声明的贡献；未声明的注册将导致激活失败。
- V1 中的清单能力声明用于审查和诊断元数据，并不是可强制执行的沙箱。产品不得将其描述为安全权限。

### 运行时生命周期

- 插件生命周期状态包括：已发现、无效、非活动、激活中、活动、失败、停用中和已禁用。
- 清单校验失败会进入 `invalid`；运行时导入或激活失败会进入 `failed`。
- 插件失败不得阻止应用其余部分或其他插件启动。
- 在可行的情况下，贡献将在首次使用时延迟激活。
- 启动种子器是 V1 中唯一可能需要在桌面启动期间激活的贡献。
- 激活是幂等的。针对同一个非活动插件的并发请求共享同一个激活 Promise。
- 停用采用尽力而为策略，并接收 AbortSignal。开始停用后，禁用插件会阻止新的贡献调用。
- 本地开发重载将停用当前实例，使用文件修改标识使模块缓存失效，校验替换后的清单，然后激活新实例。
- 宿主将为异步插件操作附加超时和 AbortSignal。V1 无法终止进程内阻塞 CPU 的代码，并会记录这一限制。

### 扩展点

- Plugin V1 将暴露四个扩展点：源导入器、工具提供者、Skill 提供者和开发种子器。
- 源导入器接收由用户或环境选择的本地源，并返回原生 Thread、一个或多个插件上下文、诊断信息以及可选的建议工作区文件名。
- 源导入器可以暴露轻量级探测操作。探测返回置信度和诊断信息，但不得修改源数据或工作区数据。
- 工具提供者将为插件上下文解析面向模型的 Tool 描述符，并通过不透明的工具引用执行明确的手动工具调用。
- 工具提供者返回标准化文本或兼容 JSON 的输出以及错误标志。丰富的二进制/资源输出延期处理。
- Skill 提供者将列出 Skill 摘要，并读取某个插件上下文中一个 Skill 的完整指令。
- 开发种子器将检查声明的环境变量，并调用已有的源导入器。它不得实现第二条导入路径。
- V1 不会暴露任意命令、菜单、设置页、模型提供者、渲染器组件或跟踪面板贡献点。

### Thread 持久化契约

- Thread 上下文将增加一个通用的插件上下文集合。
- 每个插件上下文包含稳定的上下文 ID、插件 ID、源/贡献 ID、插件上下文 Schema 版本、显示标签以及仅限 JSON 的插件自有数据。
- 插件自有数据必须通过共享的 JSON 值 Schema。不得包含函数、类、Symbol、二进制缓冲区和循环对象。
- 插件支持的 Tool 将保留通用 Tool 的名称、描述、参数和严格性字段。
- 插件支持的 Tool 还将存储插件 ID、Tool Provider ID、不透明工具引用以及可选的插件上下文 ID。
- 不会将工具专用的可执行状态复制到核心字段中。不透明引用及其关联的插件上下文就是路由契约。
- 一个插件上下文可以被多个 Tool 及其 Skill 提供者共享。
- 缺失的插件上下文或提供者不会使 Thread 的其余部分失效。Thread 仍可编辑，受影响的贡献将带诊断信息变为不可用。

### Eve 直接切换

- Plugin V1 不兼容迁移前的 Eve Thread 上下文和 Eve Tool 记录。
- 核心模型直接移除 Eve 专用上下文和 Tool 类型，只保留通用插件上下文与插件支持的 Tool。
- `@llm-space/plugin-eve/src/eve` 只返回 Eve 领域导入结果，不生成桌面持久化模型。
- `@llm-space/plugin-eve` 的插件入口负责把项目根目录、来源信息、Tools 和 Skill 作用域转换为通用插件表示。
- 新的 Eve 导入和开发种子流程只写入通用插件表示。
- Plugin V1 不保留 Eve 专用兼容读取器；迁移前的开发 Thread 需要重新导入。

### Eve 插件贡献

- Eve 插件清单将声明一个项目源导入器、一个工具提供者、一个 Skill 提供者和一个开发种子器。
- Eve 项目导入器将委托包内的 Eve 源码模块，并将其导入结果、诊断信息、Tools 和 Skill 作用域转换为通用插件契约。
- Eve 工具引用将使用相对于项目的引用或基于名称的不透明引用；绝对源路径保留在 Eve 插件上下文中，以便集中进行边界校验。
- Eve Tool Provider 将把手动调用委托给包内执行模块，并保留当前的 `toModelOutput` 行为。
- Eve Skill Provider 将把仅限项目的 Skill 列表和读取操作委托给包内 Skill 模块。
- Eve Development Seeder 将继续支持 `LLM_SPACE_EVE_PROJECT_ROOT` 和 `LLM_SPACE_EVE_THREAD_PATH`，但会通过宿主调用 Eve Source Importer。
- Eve 插件不会回退到全局 Skills 管理器。
- Eve 专用的认证、审批、沙箱和动态解析器执行仍不在本次迁移范围内。

### 通用宿主路由

- 桌面端 RPC 契约将新增通用插件请求系列，用于插件状态、源导入、Tool 列表/调用、Skill 列表/读取、启用以及本地开发重载。
- RPC 参数和结果结构由 LLM Space 所有，而不是由插件所有。
- 渲染器将传递插件 ID、贡献 ID、上下文 ID、不透明资源引用和 JSON 参数。它不会传递可执行回调，也不会导入插件包。
- 现有的渲染器 Tool 执行分发器将把插件支持的 Tool 路由到通用插件客户端。
- 模型流式处理层将继续只接收通用 Tool 描述符。除非有意包含在 Tool 名称或描述中，否则插件身份不会暴露给模型。
- 插件工具保持当前的逐步行为：模型生成 Tool Call，运行暂停，用户明确执行后填入结果。
- 无论插件元数据如何声明，V1 都禁止自动执行插件工具。

### Skill 和提示词变量路由

- 全局 LLM Space Skills 仍由现有 Skills 管理器负责。
- 对于声明了 Skill Provider 的插件上下文，Thread 通过该提供者解析 `available_skills`。
- 插件作用域的 Skill 查找绝不会回退到全局 Skills，除非所属插件明确返回宿主定义的委托结果。V1 插件不会获得此委托能力。
- 没有插件 Skill 作用域的 Thread 继续使用现有的全局 Skill 行为。
- 提示词变量代码将依赖通用 Skill 作用域解析器，而不是检查 Eve 上下文。

### 存储和诊断

- 插件设置将持久化启用状态、受信任的本地开发路径和最近一次已知的安全诊断信息。
- 每个插件将在 LLM Space 数据根目录下获得一个私有存储命名空间。
- 宿主存储 API 将阻止一个插件访问另一个插件的命名空间。
- 由于 V1 插件是受信任的进程内代码，宿主存储隔离属于 API 边界，而不是安全沙箱。
- 诊断信息将包含插件 ID、可选的贡献 ID、严重性、稳定代码、对用户安全的消息以及可选的脱敏元数据。
- 默认情况下，原始凭据、授权标头、环境变量值、提示词内容、Tool 参数和 Tool 输出不会写入持久化诊断。
- 运行时堆栈可以在开发环境中记录到本地，但未经脱敏不会发送给渲染器或分析系统。

### 插件设置界面

- 在后端契约稳定后，Plugin V1 将包含一个由宿主渲染的设置页面。
- 页面将列出插件名称、ID、版本、来源、兼容性、启用状态、生命周期状态、声明的贡献、能力声明和最新诊断信息。
- 页面可以启用或禁用插件，也可以重载本地开发插件。
- 禁用一个拥有打开中的依赖 Thread 的插件不会移除已持久化的上下文或 Tools。这些贡献在重新启用前将不可用。
- 页面不会渲染插件提供的 React 组件或任意 HTML。
- 该页面的任何实现都必须在真实的 Electrobun CEF 渲染器中验证，并在完成前进行产品设计审计。

### 版本和兼容性

- `@llm-space/plugin-api` 遵循语义化版本。
- 清单同时声明 API 版本和 LLM Space 引擎范围。
- 宿主会在运行时导入前拒绝不支持的主要 API 版本。
- 插件上下文携带独立的 Schema 版本。
- 插件可以注册一个纯上下文迁移处理器，将旧 Schema 版本迁移到当前版本。
- 上下文迁移在副本上运行，校验 JSON 输出，并且只有成功后才提交。
- 捆绑插件和宿主版本会在桌面构建期间一起测试。

### 发布步骤

- 阶段 1：引入插件 API 包、清单校验、Plugin Manager 生命周期、贡献注册表和 fixture 插件。
- 阶段 2：增加通用插件 Thread 上下文和 Tool 类型，并移除 Eve 专用持久化类型。
- 阶段 3：增加通用桌面端 RPC，以及渲染器 Tool/Skill 路由。
- 阶段 4：增加捆绑的 Eve 插件，并通过它路由 Eve 开发种子流程。
- 阶段 5：移除 Eve 专用的桌面端路由和兼容性代码。
- 阶段 6：增加由宿主渲染的插件设置页面和当前运行诊断。
- MCP、已连接的跟踪源、模型提供者和内置 Tools 在后续 RFC 决定迁移是否有价值之前，继续使用现有实现。

### 停止条件

- 如果 Eve 项目导入无法通过提议的四个扩展点表达，而必须增加 Eve 专用宿主方法，则停止实现。
- 如果通用插件状态要求在 Thread 持久化中存储非 JSON 的可执行对象，则停止实现。
- 如果无法让捆绑插件在 Electrobun 构建中实现确定性打包，则停止实现。
- 在接受远程安装或未经审查的插件代码之前，停止实现并重新审视信任模型。

## 测试决策

- 测试将断言可观察的插件行为，而不是注册表实现细节。
- 主要且最高层级的测试接缝是带有 fixture 插件的公开 `PluginManager` API。一个黑盒流程将覆盖清单发现、校验、激活、源导入、Tool 发现、作用域化 Skill 发现、明确的 Tool 执行、诊断、禁用和重载。
- fixture 插件只使用 `@llm-space/plugin-api`，证明插件作者无需导入桌面端私有模块。
- 清单契约测试将覆盖格式错误的 JSON、缺少字段、不兼容的引擎范围、不支持的 API 版本、重复插件 ID、重复贡献 ID、未声明的运行时注册和缺失的运行时入口。
- 生命周期测试将覆盖延迟激活、激活幂等性、首次使用时的并发、激活失败隔离、禁用/停用行为以及本地开发重载。
- 源导入器测试将验证探测不修改数据，并验证导入返回有效的 Thread、插件上下文、诊断信息和建议文件名。
- 工具提供者测试将验证描述符保真度、JSON Schema 保留、上下文路由、手动执行、错误标准化、取消以及缺少提供者时的诊断。
- Skill 提供者测试将验证列表/读取行为、上下文作用域以及不存在全局 Skill 回退。
- 持久化测试将通过正常 Thread Schema 序列化并重新加载插件上下文和插件支持的 Tools。
- 核心持久化测试将验证通用插件上下文和插件支持的 Tools 能正常序列化、重载，并拒绝非 JSON 插件数据。
- Eve 插件契约测试将复用现有 Eve fixtures 和示例。它们将验证指令导入、`defineTool` 参数提取、`defineSkill` 发现、仅限项目的 Skills、`toModelOutput`、手动 Tool 执行以及通过通用插件路由进行的路径边界拒绝。
- 桌面端 RPC 测试将验证插件请求类型路由到 Plugin Manager，并且渲染器模块绝不会导入插件运行时代码。
- 现有 Eve 包测试仍是 Eve 领域行为的权威测试；插件测试验证适配，而不是重复解析器内部逻辑。
- 现有的内置 Tool、MCP、全局 Skill、Thread 流式处理和手动继续行为将作为回归检查运行。
- 插件设置页面实现后，将在真实的 Electrobun CEF 渲染器中验证空、活动、已禁用、不兼容和失败插件状态；键盘行为、溢出、控制台错误以及重启后的持久化都是验收标准。
- 必要的实现验证包括插件包测试、Eve 包测试、核心和桌面端 TypeScript 检查、lint、空白字符检查、已打包构建检查，以及隔离 home 环境下的 Eve 环境导入 smoke test。

## 不在范围内

- 公开的插件市场或注册中心。
- 下载、更新或卸载远程插件包。
- 加密签名、发布者身份或供应链证明。
- 声称 V1 插件代码已被沙箱化，或在插件不受信任时可以安全运行。
- 操作系统级进程沙箱、基于能力安全的模块加载或受限 Bun 导入。
- 任意渲染器 UI、React 组件、HTML、CSS、菜单、命令或导航贡献。
- 在模型流式处理中自动执行插件 Tools。
- 丰富的二进制 Tool 结果、资源、附件或流式 Tool 输出。
- 插件提供的模型提供者。
- 插件提供的跟踪面板或通用跟踪源迁移。
- 在本 RFC 中将 MCP、内置 Tools、全局 Skills、模型提供者或 Langfuse 支持迁移为插件。
- Eve 运行时/会话 API、Eve 跟踪、Eve Agent Runs、动态 Eve 解析器、Eve 审批流程、Eve 认证或 Eve 沙箱集成。
- Eve 插件上下文中的全局 Skill 回退。
- 迁移或读取 Plugin V1 之前生成的 Eve 专用 Thread 数据。
- 修改导入的外部项目源代码。
- 插件之间的直接导入，或访问其他插件的私有存储。
- 在桌面端 Bun 进程退出后仍继续运行的后台插件工作进程。

## 补充说明

- Plugin V1 有意优先验证架构和支持受信任的本地开发。Eve 迁移是验收案例，而不只是示例。
- 产品文案和文档必须明确说明信任边界。进程内插件可以直接导入 Bun/Node API；清单能力声明无法阻止这种行为。
- 未来的不受信任插件 RFC 应评估子进程隔离、操作系统沙箱配置、签名制品、网络中介和更窄的消息协议。这些工作不得被暗示为 V1 的能力。
- VS Code 的清单/贡献分离、Dify 的清单声明插件能力以及 MCP 可发现的 Tool 契约，都是有用的先例。LLM Space 应采用其中稳定的理念，而不是复制它们完整的产品表面。
- 北极星验收应端到端衡量：一个 Eve 项目能够进入可用的插件支持 Thread，并在没有任何 Eve 专用桌面端/核心路由参与的情况下完成一次手动 Tool 调用。
- 当实现细节不一致时，本 RFC 取代 kaizen 日志中仅用于规划的插件架构摘要；kaizen 日志仍保留产品决策历史。
