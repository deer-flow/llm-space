§ 00 — 框架能力 × Windows UI 规范

框架给了什么，规范要求什么。

先分清三类事实：Electrobun 已经替我们做掉的、它做不了需要我们自接的、以及 Windows 用户肌肉记忆强制要求的。

Electrobun 就绪面直接用
框架内置，Windows 上开箱可用
  • ●WebView2 系统内核（Chromium 系，Win11 自带 runtime）
  • ●自动更新全链路：update.json + tar.zst + bsdiff 补丁（1.18.1 修复）
  • ●产物命名 {channel}-win-x64-*，与现有 updates feed 模型一致
  • ●自解压 Setup.exe 安装器（打包进 -Setup.zip）
  • ●build.win.icon 指 .png 自动转 .ico
  • ●titleBarStyle 跨平台；1.18 修复无边框 resize 边框/DWM 阴影/Aero Snap
Electrobun 受限面要自接
框架缺口，需要我们绕行或补齐
  • ✕Windows 代码签名：CLI 无 signtool 集成（mac 签名是内置的）→ hook 自接
  • ✕setPageZoom / getPageZoom：文档标注 Windows no-op → 渲染层兜底
  • +菜单 accelerator 仅可靠支持 Ctrl+单字符；Ctrl+Shift+P 类不保证
  • +trafficLightOffset / hiddenInset 语义在 Windows 退化为 hidden
  • +无交叉编译：必须 Windows runner 本机构建
  • +URL scheme / 文件关联 Windows 未支持（本应用未用，无影响）
Windows UI 硬规范必须守
用户肌肉记忆级约定（Fluent + 主流工具惯例）
  • +窗口控制在右侧：最小化/最大化/关闭，关闭键 hover 红
  • +自绘标题栏须保住：双击最大化、右键系统菜单、Snap Layouts、贴边分屏
  • +Ctrl 替代 Cmd；F11 全屏；UI 显示 Ctrl+ 而非 ⌘
  • +文案：文件资源管理器 / 回收站（不再是 Finder / Trash）
  • +未签名 exe 触发 SmartScreen 拦截页（签名+装机量积累信誉后消失）
已达标 / 明确不做零成本
现有实现直接过关，或收益趋零主动排除
  • ●深色主题 + 自绘细滚动条（WebView2 同为 Chromium，::-webkit-scrollbar 生效）
  • ●字体栈 "Geist Variable", system-ui（Segoe UI 自动兜底）
  • ●高 DPI：WebView2 自动处理
  • ✕Mica/Acrylic 材质：框架未暴露 DWM API，且应用为不透明深色，主动排除
  • ✕Jump List / 托盘 / 系统通知：当前功能面不需要

§ 03 — 落地路径

勾选确认后，粘回对话。

本页只阐述方案和收集决定。粘回对话后由 Claude 按阶段执行——每阶段独立可验证、可回滚。

P0 → P1

先证明能跑，再证明能发

P0 在 Windows 11 实机/VM 上跑通开发链路并实测三个不确定点（setPageZoom、无边框 Snap Layouts、WebView2 渲染差异）——A2/F 两个决策的最终确认依赖这里的实测结果，A1 始终作为回退项保留。P1 让 CI 产出第一个可安装、可自动更新的未签名 canary。

P2 → P3

先好用，再好看官方

P2 把窗口 chrome、菜单/keymap、内置工具、缩放兜底、全部视觉细节收掉，达到与 mac 等价的日常可用。P3 处理签名（含签名与补丁哈希的时序验证）、文档口径与对外 release notes，升级 stable 通道。

风险

四个已知风险与对策

① Windows 验证环境是 P0 硬前置；② 自绘标题栏依赖 Electrobun 1.18 的新代码，成熟度未知——P0 实测后再终审 A2，留 A1 回退；③ Windows 签名与 updater 补丁哈希的时序（C2 前专项研究）；④ WebView2 与 mac WKWebView 渲染差异（双方都对 Chromium 友好，风险低但 P0 过一遍）。
